Weekly Web 架构设计文档 V1.0

1. 文档信息

项目名称 Weekly Web
项目定位 基于 ruanyf/weekly Markdown 内容的开源科技周刊阅读平台
对应 PRD Weekly Web 产品需求文档（PRD）V1.0
核心技术栈 Next.js 全栈、SQLite、better-sqlite3、GitHub Actions
版本 V1.0

2. 架构目标

  • 支持全部历史周刊的 Web 阅读、目录导航、章节定位和响应式展示。
  • 支持全文搜索、搜索结果定位和关键词高亮。
  • 支持用户收藏、阅读进度、阅读历史和主题偏好。
  • 页面体验保持简约化、低噪声和 Apple 风格高级感，优先服务阅读效率。
  • 保持内容源简单可靠：周刊正文继续来自仓库 Markdown 文件。
  • 通过 GitHub Actions 同步上游内容、生成构建期 JSON 和搜索索引。
  • 通过 GitHub Actions 完成自动检查、构建和部署。

3. 架构边界

  • V1 不建设后台 CMS，不提供在线编辑周刊正文。
  • V1 不把周刊正文、章节、图片、外链、标签作为主数据写入 SQLite。
  • SQLite 只保存少量用户态数据：用户、收藏、阅读进度、阅读历史、主题偏好。
  • 内容元信息、页面标签、分类和搜索索引都由 Markdown 构建期解析生成。
  • V1 搜索使用构建期 search-index.json + Fuse.js + Next.js 服务端搜索，不引入 Elasticsearch、Meilisearch 或数据库全文检索。
  • 登录不作为主导航页面暴露，只在收藏、个人中心等需要登录的功能中以弹窗触发。
  • 评论、RSS、纠错反馈、AI 问答、推荐系统、会员和付费能力不进入 V1。

5. 架构决策

问题 V1 决策 原因
周刊正文是否入库 不入库，继续以 Markdown 文件为内容源。 当前内容已经由 Git 管理，入库会增加同步、迁移和一致性成本。
页面标签是否入库 不入库，由构建期解析生成。 V1 没有人工运营标签，自动标签作为展示和筛选字段即可。
搜索技术 search-index.json + Fuse.js + Next.js 服务端搜索。 约 400 篇 Markdown 数据量小，Fuse.js 足够支持关键词、模糊匹配和结果排序。
数据库职责 SQLite 只保存用户态数据，Node 侧通过 better-sqlite3 访问。 V1 只需要保存少量收藏、进度、历史和设置，SQLite 足够轻量，避免引入独立数据库服务；better-sqlite3 同步 API 简单，适合 Route Handler 内短事务读写。
内容同步 由 GitHub Actions 定时或手动同步上游周刊内容。 内容源来自 GitHub，使用 Actions 拉取、解析和生成索引，部署链路最短。
登录方式 V1 只支持 GitHub 和 LinuxDo 登录。 登录能力仅用于区分游客和登录用户，并同步用户态数据。
部署方式 GitHub Actions 负责检查、构建和部署。 内容和代码都在 GitHub，CI/CD 路径短。

6. 技术选型

层级 选型 说明
前后端框架 Next.js App Router 同一项目内完成页面、API、服务端渲染和静态生成。
数据库 SQLite + better-sqlite3 保存少量用户相关数据，不保存周刊正文主数据；运行时通过 Node 进程访问 SQLite 文件。
内容源 Markdown 文件 直接复用现有 docs/issue-*.md，降低同步和迁移复杂度。
搜索 构建期搜索索引 + Fuse.js V1 数据量较小，使用 Fuse.js 查询构建期索引，暂不上独立搜索服务。
部署 GitHub Actions 在定时任务、手动触发、推送或合并后执行内容同步、检查、构建和部署。

  ↓

  ↓
生成每期渲染数据 issue-<number>.json
  ↓
生成搜索索引 search-index.json
  ↓

  ↓
Next.js 页面读取构建产物渲染
  • README.md 可作为首页年份分组的参考，但不应成为唯一数据源。
  • 详情页根据期号读取对应 Markdown 解析产物。
  • 目录由 Markdown 二级、三级标题自动生成。
  • 外链统一设置新标签页打开，避免打断阅读进度。

8. GitHub Actions 内容同步方案

V1 的内容同步不依赖后台 CMS 或人工入库流程，而是由 GitHub Actions 自动完成。 Actions 负责拉取或检测 ruanyf/weekly 内容变化，解析 Markdown，并生成前端可直接读取的静态 JSON 产物。 

schedule / workflow_dispatch / push
  ↓
拉取上游 ruanyf/weekly 或检查本仓库 docs/issue-*.md
  ↓
比对新增、修改、删除的周刊 Markdown
  ↓
解析 Markdown，生成 issues.json、issue-<number>.json、issue_index.json
  ↓
生成 search-index.json
  ↓
执行格式校验、链接校验、构建校验
  ↓
提交构建产物或触发部署
  • 同步任务支持定时触发，适合跟随周刊发布节奏自动更新。
  • 同步任务支持 workflow_dispatch 手动触发，便于临时修复或重新生成索引。
  • 内容解析产物进入版本管理或构建产物目录，运行时不需要扫描 Markdown。
  • SQLite 用户数据不参与内容同步，避免 Actions 覆盖线上用户态数据。
  • 当 Actions 检测到内容源无变化时，应跳过索引提交和部署触发。
  • 如果同步任务自动提交构建产物，提交范围只允许包含 Markdown 内容、构建期 JSON、搜索索引和相关清单文件。
  • 如果采用部署时生成构建产物，Actions 应在构建阶段生成临时产物，不需要把产物提交回仓库。
  • SQLite 文件、备份文件和迁移运行状态必须加入部署环境持久化存储，不应进入 Git 仓库。

9. 标签与分类策略

  • V1 不提供人工标签后台，不维护标签表。
  • 页面标签来自构建期派生，包括年份、期号、章节类型和关键词规则。
  • 章节类型优先使用周刊内已有标题，例如 文章、工具、AI 相关、资源、图片、文摘、言论。
  • 关键词标签只用于展示和筛选，不作为权限、推荐或运营配置依据。
  • 如果后续需要人工精选专题，再新增 collections 或 tag_overrides 配置文件，而不是直接改数据库结构。

10.1 页面整体逻辑

  • 首页只承载两个动作：进入最新一期周刊、输入关键词搜索。
  • 周刊详情页是主阅读页，默认打开最新一期，并允许通过年份、月份、周切换到其他期数。
  • 收藏不再单独占据主入口，而是作为详情页和搜索结果中的上下文操作。
  • 个人中心承载登录后的用户态内容，包括最近阅读、收藏和主题设置。
  • 登录不占主导航位，未登录时在收藏或个人中心中弹出登录，降低入口噪声。

10.2 页面视觉风格

  • 整体风格采用简约化、低噪声、高留白的高级感视觉方向，接近 Apple 式克制体验。
  • 页面主色控制在中性灰、白和单一蓝色强调色，避免多色卡片和强装饰背景。
  • 布局强调内容主次关系，首页与详情页都优先让主内容获得完整视线焦点。
  • 控件使用细边框、轻阴影和柔和圆角，减少厚重卡片感和视觉堆叠。
  • 交互反馈保持轻量，避免动画和渐变过度抢占阅读注意力。

11. 数据结构设计

V1 数据库只保存用户态数据，不创建 issues、sections、tags、links 等内容主表。 这些内容都来自构建产物，用户表只通过 issue_number 和 section_id 引用内容位置。 用户唯一性由 provider + provider_account_id 保证。

11.0 SQLite 运行约束

  • Node.js 运行时使用 better-sqlite3 连接 SQLite，不使用独立数据库服务。
  • SQLite 文件路径通过环境变量配置，例如 SQLITE_DB_PATH。
  • 生产环境必须部署在具备持久化磁盘的 Node.js 运行环境，或挂载持久化卷保存 SQLite 文件。
  • 初始化时开启 WAL 模式和必要的外键约束，提升读写稳定性。
  • 所有写操作保持短事务，避免长时间占用数据库锁。
  • schema 迁移通过版本化 SQL 文件执行，部署前先备份 SQLite 文件。
  • GitHub Actions 不读写线上 SQLite 文件，只负责内容同步、索引生成、检查、构建和部署。

11.1 原型反推的构建期数据结构

原型中的周刊切换、最新一期、搜索定位和目录渲染，都应由构建期 JSON 产物驱动。 这些结构属于只读内容数据，不写入 SQLite。

12. API 设计

所有需要访问 SQLite 的接口必须运行在 Node.js runtime，不能部署到 Edge runtime。 Route Handler 内通过 better-sqlite3 执行短事务读写。

接口 用途
GET /api/issues 返回最新一期和周刊索引数据，支持首页最新一期和详情页筛选器。
GET /api/issues/[number] 返回指定期号的详情渲染数据、目录和章节锚点。
GET /api/search?q=keyword 返回搜索结果、命中片段和定位锚点。
GET /api/me/favorites 获取当前用户收藏列表。
POST /api/me/favorites 收藏整期周刊或具体条目。
DELETE /api/me/favorites/[id] 取消收藏。
GET /api/me/progress/[number] 获取某期阅读进度。
PUT /api/me/progress/[number] 更新某期阅读进度。
GET /api/me/history 获取阅读历史。
GET /api/me/settings 获取主题等用户配置。
PUT /api/me/settings 更新用户配置。

13. 登录态与安全边界

  • V1 只支持 GitHub 和 LinuxDo 登录，不提供邮箱密码注册。
  • 登录入口以全局弹窗形式出现，不设置独立登录主页面和主导航登录项。
  • 游客可浏览、搜索、切换主题，并将阅读进度和主题偏好保存到本地。
  • 登录用户可同步收藏、阅读进度、阅读历史和主题偏好。
  • 游客点击收藏、进入个人中心时触发登录弹窗；登录成功后继续当前上下文。
  • 所有 /api/me/* 接口必须从服务端登录态获取 user_id，不得信任客户端传入的用户 ID。
  • 用户身份以 provider + provider_account_id 唯一确定。
  • 收藏、阅读进度、阅读历史和用户设置查询都必须带 user_id 条件。
  • 写接口启用 CSRF 防护或同等的 SameSite Cookie + 服务端校验策略。
  • 数据库对 user_id、issue_number、section_id 建立必要索引和唯一约束，避免重复收藏和串号读取。
  • SQLite 文件权限限制为应用进程可读写，部署目录不得暴露数据库文件下载路径。

14. 搜索方案

  • 搜索技术固定为构建期 search-index.json + Fuse.js + Next.js Route Handler 服务端搜索。
  • 索引粒度为章节或条目，不以整篇周刊作为唯一搜索单元。
  • 索引字段包含 issue_number、issue_title、section_title、anchor、text、tags。
  • Fuse.js 查询字段优先级：标题和章节标题高于正文，标签高于普通正文。
  • 搜索结果返回期号、周刊标题、章节名称、命中片段、标签、匹配分数和跳转锚点。
  • 关键词高亮根据 Fuse.js 匹配结果生成，页面只负责展示。
  • 当索引文件超过 10MB、P95 搜索响应超过 300 毫秒，或需要拼音搜索/语义搜索时，再评估 SQLite FTS、Meilisearch 或其他独立搜索服务。

15. 阅读进度方案

  • 游客阅读进度保存到 localStorage。
  • 登录用户阅读进度写入 SQLite。
  • 进度记录采用滚动百分比和最近章节锚点双字段。
  • 外链默认新标签页打开，避免离开详情页导致上下文丢失。

16. 部署流程

schedule / workflow_dispatch / push / pull request
  ↓
GitHub Actions
  ↓
安装依赖
  ↓
同步上游 Markdown 内容
  ↓
类型检查 / Lint / 测试
  ↓
解析 Markdown 并生成内容索引
  ↓
Next.js build
  ↓
部署到目标平台
  • 部署目标可以是具备持久化磁盘的 Node.js 服务器、Docker 容器或支持 SQLite 持久化的托管环境。
  • 如果部署到无持久化文件系统的平台，应改用 SQLite 兼容托管服务或挂载持久化存储，不能把用户态数据写入临时文件系统。
  • GitHub Actions 只同步内容源和构建产物，不同步、不覆盖线上 SQLite 用户数据。
  • SQLite schema 迁移在部署前执行，失败则停止部署；迁移前应备份数据库文件。
  • 生产环境通过环境变量注入 SQLite 文件路径、认证密钥和 OAuth 配置。
  • 使用 Docker 部署时，SQLite 文件目录必须挂载 volume，镜像本身不携带生产数据库。
  • better-sqlite3 是 Node 原生依赖，CI 构建环境和部署运行环境应保持平台一致，或在目标镜像内安装依赖。

17. V1 实施顺序

  1. 搭建 Next.js 项目和基础页面路由。
  2. 实现 Markdown 扫描、解析和内容索引生成。
  3. 实现首页最新一期、搜索入口、周刊详情页、目录导航、详情页年份/月/周筛选和外链新标签页。
  4. 实现搜索索引和搜索页。
  5. 接入 SQLite、better-sqlite3、schema 迁移、OAuth 登录和全局登录弹窗。
  6. 实现收藏、阅读进度、阅读历史和主题设置。
  7. 配置 GitHub Actions 内容同步、构建和部署。

18. 后续演进

  • 当需要在线编辑、审核流或内容版本管理时，再引入 CMS 或正文入库。
  • 当搜索索引过大或响应超过 300 毫秒时，再升级搜索实现。
  • 当用户态数据增长到单机 SQLite 难以承载，或需要多实例高并发写入时，再评估 PostgreSQL 等独立数据库。
  • 当访问量增大时，为详情页和搜索接口增加缓存策略。
  • 当需要内容增长能力时，再考虑 RSS、纠错反馈、推荐和 AI 问答。
