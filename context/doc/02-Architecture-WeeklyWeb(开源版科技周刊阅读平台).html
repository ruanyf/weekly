Weekly Web 架构设计文档 V1.0

1. 文档信息

项目名称 Weekly Web
项目定位 开源版科技周刊阅读平台
工作目录 web
内容来源 仓库根目录下的 docs 周刊内容

2. 技术架构

技术 用途
Next.js Web 应用主框架，负责页面路由、服务端能力和应用构建。
UnoCSS 原子化样式方案，承载页面布局、响应式和视觉风格。
better-sqlite3 Node.js 访问 SQLite 的数据库驱动，用于保存少量用户态数据。
Fuse.js 前端或服务端轻量全文搜索，基于静态生成阶段产出的索引文件查询。
Vite 前端开发和工具链能力，用于原型、辅助脚本或独立构建场景。
JavaScript 项目主要开发语言。

3. 核心原则

  • 一切以性能优先，页面访问阶段尽量减少计算、数据库读取和 Markdown 解析。
  • 内容处理方式参考 Hugo 静态博客生成：先生成静态产物，再提供页面访问。
  • 周刊正文不进入 SQLite 内容表，运行时也不承载正文主数据。

4. 目录约束

  • 网站应用代码统一放在 web 目录下。
  • web 内部维护页面、组件、样式、内容生成脚本和 SQLite 访问代码。
  • 仓库根目录下的 docs 作为周刊原始内容目录，不移动到 web 内。
  • web 内提供内容生成脚本，从根目录 docs 读取并生成静态页面、JSON 数据和搜索索引。

5. 内容生成约束

docs 目录会通过 GitHub Actions 定时同步更新。同步后必须重新执行内容生成流程，将 Markdown 预解析为静态产物； 运行时不扫描 docs、不解析 Markdown，只读取已经生成的静态产物。 

GitHub Actions 同步 docs / 构建触发
  ↓
web 内容生成脚本读取 ../docs
  ↓
解析期号、标题、日期、封面图、目录、章节、正文、外链
  ↓
生成静态 HTML、JSON 数据与 Fuse.js 搜索索引
  ↓
Next.js 构建或运行时读取静态产物
  • 解析逻辑应放在 web 内部，由内容生成脚本调用。
  • 解析结果必须落地为静态产物，避免运行时重复解析，也避免维护冗余持久化内容表。
  • 运行时优先读取静态 HTML、JSON 和索引文件，保证页面访问路径足够轻。
  • SQLite 只保存用户态数据，不保存周刊正文、章节、图片、外链等内容主数据。
  • GitHub Actions 完成 docs 同步后，触发内容生成与构建流程。
  • 当 docs 同步后，必须重新执行静态生成流程，确保最新内容被写入静态产物。

6. 数据边界

  • 周刊内容数据来自 docs 生成出的静态产物。
  • 搜索数据来自静态生成阶段产出的 Fuse.js 索引。
  • 用户收藏、阅读进度、阅读历史、主题偏好等少量用户态数据保存到 SQLite。
  • SQLite 通过 better-sqlite3 接入，不引入独立数据库服务。
  • 运行时不得扫描 docs 或重新解析 Markdown；除用户状态外，不新增内容持久化表。

7. 页面与交互约束

  • 页面必须采用响应式布局，兼容桌面端、平板端和移动端。
  • 首页保持简洁，只突出最新一期周刊和搜索入口。
  • 周刊详情页默认展示最新一期，并支持通过年份、月份、周筛选其他周刊。
  • 搜索使用静态生成的 Fuse.js 索引，对标题、章节和正文片段进行关键词匹配。
  • 收藏、阅读历史等需要登录的能力在用户触发时再展示登录入口。

8. 样式 & 风格

  • 整体视觉保持简约现代化。
  • 风格参考 Apple：高留白、低噪声、清晰层级、克制色彩。
  • 避免复杂装饰、过多卡片、强渐变和嘈杂信息密度。
  • 优先保证阅读体验，正文区域要清晰、安静、易长时间阅读。
  • 交互控件使用统一圆角、细边框和轻量反馈。
